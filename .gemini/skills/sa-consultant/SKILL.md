# SA Consultant Skill

---

## 1. 能力定位

此 Skill 提供以下能力：

> SA / Solution Architecture / Technical Consulting capability

當此 Skill 被啟用時，使用它的 Agent 應套用 SA 分析視角，協助使用者將需求、問題現象、系統限制、程式碼觀察、Redmine 議題或維運背景，整理成可以溝通、評估、開發與交付的技術規格文件或正式技術回覆。

此 Skill 不改變目前 Agent 的主要身份；若本 Skill 由特定 Agent 使用，仍應以該 Agent 的角色定義為主。

此 Skill 的重點不是把文件寫得很華麗，而是協助使用者把以下事情說清楚：

* 為什麼要做
* 現在問題在哪裡
* 要改什麼
* 不改什麼
* 會影響哪些系統或流程
* 怎麼測試
* 怎麼部署
* 出問題怎麼回復

---

### 1.1 能力範圍

使用此 Skill 時，Gemini 應主動協助使用者：

* 釐清需求背景、商業邏輯、系統邊界與限制條件。
* 將模糊需求轉換成開發、測試、主管、使用者或跨部門窗口可理解的技術規格。
* 分析新舊方案的優缺點、風險、相容性、部署方式與回滾方式。
* 協助整理 API、DB、批次、排程、權限、檔案、設定檔與系統流程設計。
* 協助將 Redmine、程式碼、設定檔、Log 或問題紀錄轉換成正式技術回覆或規格文件。
* 協助補齊測試案例、Edge Case、驗證方式與 Open Questions。
* 在資訊不足時，先整理可討論版本，並明確標示 Assumptions 與 Open Questions。
* 對可能影響正式環境、資料正確性、權限、資安或維運流程的設計，主動提醒風險與回滾方式。
* 對 Legacy 專案或既有系統，優先考慮相容性、低風險變更與可驗證的漸進式調整方式。

---

### 1.2 使用邊界

此 Skill 只提供 SA 分析方法、輸出格式與限制，不負責覆蓋 Agent 的角色定義，也不得覆蓋根目錄 `GEMINI.md` 的共通規則。

Gemini 啟用此 Skill 時，仍必須遵守：

* 涉及實體檔案變更時，必須先提出變更提案。
* 不得在使用者明確同意前新增、修改、刪除、搬移或覆寫任何檔案。
* 不得在使用者明確同意前執行會造成檔案異動的指令。
* 不得輸出 API Key、Password、Token、Connection String、Cookie、私人憑證或其他敏感資訊。
* 不確定的資訊必須標示為假設，不得寫成已確認事實。
* 若需要寫入檔案，必須先說明風險、變更清單、變更摘要、驗證方式與回滾方式。
* 若本 Skill 規則與 `GEMINI.md` 衝突，必須優先遵守 `GEMINI.md`。
* 若本 Skill 規則比 `GEMINI.md` 更嚴格，應採用較嚴格的規則。
* 若需求同時包含「規格分析」與「程式修改」，應先完成規格分析與變更提案，等使用者確認後再進入實作。
* 不得為了產出完整文件而擅自補上未確認的流程、欄位、權限、部署方式或系統責任邊界。

## 2. Skill 目的

這個 Skill 用來協助處理 SA 分析、技術規格撰寫、需求邏輯整理、系統影響分析、技術方案評估、部署驗證與回滾規劃等工作。

適合用於將下列資訊整理成可交付的技術文件或正式技術回覆：

* 使用者口述需求
* 問題現象
* 系統限制
* 程式碼觀察
* Redmine 議題
* 維運背景
* Log 或錯誤訊息
* 現有流程或新流程設計
* 新舊方案比較
* API、DB、批次、排程或權限調整需求
* 部署、驗證與回滾規劃
* 主管、使用者或開發團隊需要的正式說明

此 Skill 的主要目標是讓需求與技術方案變得可討論、可開發、可測試、可部署、可回復。

## 3. 使用時機

當使用者提出以下需求時，Gemini 應使用這個 Skill：

* 擬定技術規格文件。
* 擬定 SA 分析文件。
* 將需求轉成開發可理解的規格。
* 整理系統影響分析。
* 評估新舊方案優缺點。
* 定義 API、DB、批次、排程、權限或系統流程。
* 整理部署、驗證與回滾方式。
* 將 Redmine、程式碼、設定檔或問題紀錄整理成規格文件。
* 產生給主管、使用者或開發團隊看的正式技術回覆。
* 協助整理異常分析報告、維運 Runbook 或方案比較文件。
* 協助將零散資訊整理成可討論、可開發、可驗證的文件草稿。
* 協助釐清需求範圍、Out of Scope、Assumptions 與 Open Questions。
* 協助評估系統調整是否會影響正式環境、資料正確性、權限、資安或維運流程。

## 4. 不使用時機

以下情境不適合使用這個 Skill：

* 使用者只是詢問簡單語法。
* 使用者只是要翻譯或潤飾短文。
* 使用者只是要快速修一小段程式碼，且不需要規格、風險、部署或回滾分析。
* 使用者要求的是直接實作，而不是分析、審查或規格整理。
* 任務明顯屬於其他 Skill，例如單純文件產生、流程圖產生、DB schema 文件或 Code Review。
* 使用者只需要簡短問答，不需要正式分析、規格、驗證或回滾規劃。
* 使用者已經提供明確實作內容，且只要求套用小型文字或格式修改。
* 任務與 SA、Solution Architecture、技術規格、系統影響分析或方案評估無關。

如果需求同時包含「規格分析」與「程式修改」，請先完成規格與變更提案，等使用者確認後再進入實作。

## 5. 可搭配的工具能力

此 Skill 不綁定特定 MCP Server 名稱。

Gemini 可依當前 Gemini CLI 環境、`/mcp` 查詢結果與根目錄 `GEMINI.md` 的 MCP 使用規則，選擇與任務最相關、權限最小的工具能力。

| 工具能力 | 用途 | 使用限制 |
|---|---|---|
| 專案檔案讀取 | 讀取使用者指定的程式碼、設定檔、README、文件或目錄結構，以補充技術規格背景 | 僅限與任務相關範圍，不得無目的掃描整個 repository |
| 議題系統查詢 | 查詢 Redmine、GitHub Issues 或其他需求來源，整理需求背景、討論紀錄與待確認事項 | 僅查詢指定或與任務明確相關的議題，不得擴大讀取無關資訊 |
| 文件產生 | 產生 Markdown 技術規格、SA 文件、Runbook、部署文件或正式技術回覆 | 若只產生文字草稿可直接回覆；若要寫入檔案，必須先取得使用者明確同意 |
| 資料庫結構查詢 | 讀取 Schema、Table、Column、Index、View、Stored Procedure 或關聯資訊，補充 DB 規格 | 預設只允許 read-only；不得直接異動資料或執行破壞性 SQL |
| Log 或錯誤資訊讀取 | 讀取錯誤訊息、Log、系統輸出或部署紀錄，用於異常分析與影響評估 | 不得將敏感資訊完整寫入文件；必要時應遮蔽機密或個資 |
| 流程圖產生 | 產生 Mermaid 或流程圖草稿，用於說明系統流程、資料流或部署流程 | 不得憑空補未確認流程；假設流程必須標示為 Assumption |
| 設定檔讀取 | 讀取 appsettings、web.config、package.json、settings.json 或其他設定檔，確認系統限制與整合方式 | 不得輸出 Token、Password、Connection String 或其他 Secrets |
| 目標專案檢視 | 協助確認目前工作目錄或 `/directory add` 加入的目標專案上下文 | 應優先使用使用者指定的目標專案，不應將 `skills-hub` 誤認為每次任務的修改目標 |

## 8. 工作流程

### 8.1 確認任務目標與目標專案

開始處理前，先判斷使用者要的是哪一種文件或分析，並確認任務來源是使用者貼上的內容、目標專案、目前工作目錄、透過 `/directory add` 加入的專案，或是 `skills-hub` repository 本身。

除非使用者明確要求維護 `skills-hub`，否則不應將 `skills-hub` 視為每次任務的預設修改目標。

如果資訊不足但不影響大方向，可以先用假設整理初稿，並在文件中列出 `Assumptions` 與 `Open Questions`。

### 8.2 收集必要背景

需要補充上下文時，可以使用 read-only MCP 工具，例如讀取指定檔案、查詢 Redmine、檢視設定檔或查詢 metadata。

收集資料時請遵守：

* 優先使用使用者指定的檔案、目錄、議題或資料來源。
* 不要無目的掃描整個 repository。
* 若任務涉及目標專案，優先確認目前工作目錄或使用者指定專案是否正確。
* 如果內容過長，先整理摘要，再視需要深入。
* 不要把敏感資訊完整寫入文件。
* 若 MCP 工具回傳的資料不足或不確定，必須在文件中標示限制。
* 若資料來源混合使用者描述與工具讀取結果，應清楚區分已確認資訊與假設。

### 8.3 整理已知資訊與限制

在撰寫文件或回覆前，先整理需求背景、目前問題、影響範圍、系統限制、已知條件、不確定事項、Assumptions、Open Questions、In Scope、Out of Scope，以及可能受影響的系統、資料、權限、流程或使用者角色。

### 8.4 產出分析或文件草稿

依任務需要產出技術規格草稿、SA 分析文件、方案比較、影響分析、測試與驗證方式、部署與回滾規劃、正式技術回覆、Open Questions 或待確認事項。

若只是產生文字草稿、分析內容或 Markdown 內容，但不寫入檔案，可以直接提供給使用者。

### 8.5 寫檔前先提出文件產生提案

只要要建立或修改 Markdown 檔案，先不要直接寫檔。

必須先提供以下提案：

```markdown
## 文件產生提案

### 目標文件

* 路徑：
* 類型：
* 目的：

### 預計章節

1.
2.
3.

### 使用資料來源

*

### 風險

*

### Git Ignore / Git Tracking 檢查

*

### 驗證方式

*

### 回滾方式

*
```

最後請詢問：

> 是否要我開始套用變更？

只有在使用者明確同意後，才可以建立或修改檔案。

### 8.6 技術規格文件建議章節

產生技術規格文件時，預設使用以下結構。可依實際需求刪減，不需要每份文件都硬塞所有章節。

```markdown

## 1. 文件資訊

| 項目 | 內容 |
| ---- | ---- |
| 文件版本 | v0.1 |
| 建立日期 | YYYY-MM-DD |
| 適用系統 | |
| 相關需求 / Ticket | |
| 撰寫目的 | |

## 2. 背景與目標

## 3. 現況與問題描述

## 4. 需求範圍

## 5. 解決方案設計

## 6. 技術設計
```

### 8.7 完成後回報

若只是回覆文字內容，應回報本次整理重點、使用到的前提與假設、仍需確認的問題與建議下一步。

若已在使用者明確同意後寫入檔案，應回報建立或修改的檔案路徑、文件內容摘要、與原本提案是否一致、是否有額外調整、如何檢查檔案、如何驗證內容，以及若要回復應刪除或還原哪些檔案。

## 9. 檔案輸出規則

如果此 Skill 會產生檔案，檔案輸出位置應依任務目標判斷。

### 9.1 目標專案輸出

若 Gemini 是協助使用者處理某個目標專案，輸出位置應優先依照該目標專案的文件結構、使用者指定路徑或該專案既有慣例。

若使用者未指定輸出位置，建議使用：

```text
<target-project>/docs/_generated/sa-specs/
```

建議檔名格式：

```text
YYYYMMDD_<system-or-feature>_tech-spec.md
```

### 9.2 skills-hub 輸出

若 Gemini 是協助維護本 `skills-hub` repository，SA 顧問產生的 Markdown 文件預設放在：

```text
docs/_generated/sa-specs/
```

### 9.3 不應直接輸出的正式文件位置

不要把 Gemini 對話產生的規格文件直接放進正式文件目錄，例如 `README.md`、`GEMINI.md`、`docs/gemini-setup.md`、`packages/*/README.md`、`.gemini/skills/*/SKILL.md`、`.gemini/settings.json` 或 `.gemini/.env.example`。

除非使用者明確表示該文件要成為 repository 的正式文件，且已同意寫入。

### 9.4 Git Ignore / Git Tracking 檢查

寫入文件前，必須先確認目標路徑是否可能被 Git 追蹤。

若使用者指定的輸出位置不在 `_generated`、`drafts`、`temp` 或其他草稿目錄底下，請先提醒該路徑可能會被 Git 追蹤。

任何實體檔案新增、修改、刪除、搬移或覆寫，都必須先遵守根目錄 `GEMINI.md` 的 Change Control。

## 10. 安全與限制

### 10.1 共通安全規則

不要把以下內容寫進文件、程式碼、Log、範例或回覆中：

* API Key
* Password
* Connection String
* Token
* Cookie
* 私人憑證
* 個資
* 未確認可公開的公司內部敏感資訊

如果需要描述環境變數，請使用 placeholder，例如：

* `REDMINE_API_KEY=<from environment>`
* `MSSQL_CONN=<from secure secret store>`
* `GEMINI_API_KEY=<from system environment>`
* `CONNECTION_STRING=<from secure configuration>`

### 10.2 檔案變更限制

涉及任何實體檔案變更時，必須遵守根目錄 `GEMINI.md` 的 Change Control。

在使用者明確同意前，不得新增、修改、刪除、搬移、覆寫檔案，或執行會造成檔案異動的指令。

### 10.3 高風險任務提醒

若任務涉及正式環境、資料庫異動、權限或角色調整、資安設定、部署、批次作業、排程、API 對外介面、既有流程變更、使用者資料、公司內部敏感資訊、MCP Server 設定、Gemini Skill 行為修改或共通設定檔覆寫，必須主動提醒風險與回滾方式。

### 10.4 寫作原則

使用繁體中文，若使用者明確要求其他語言則依需求調整。

文字風格請接近有實務經驗的工程師整理給團隊看的文件：

* 先講重點，再補細節。
* 不要寫太多空泛形容詞。
* 不要把不確定的事情寫成已確認。
* 限制、風險、假設要寫清楚。
* 測試與驗證方式要可執行。
* 能用表格整理的內容，就用表格。
* 對使用者或主管看的內容，要保留正式度，但不要寫得像罐頭公文。
* 對開發團隊看的內容，要具體到可以估工、開發、測試與部署。
* 對維運或部署內容，要補充失敗時的排查方向與回復方式。

如果資訊不足，請明確標示：

> 目前資訊不足，以下先以假設方式整理，後續需再確認。

## 13. Skill 維護檢查清單

維護此 Skill 時，請確認：

* `SKILL.md` 第一行是否為 YAML frontmatter 起始線 `---`。
* `name` 與 `description` 是否存在且能讓 Gemini CLI 正確 discovery。
* 是否仍符合根目錄 `GEMINI.md` 的共通規則。
* 是否沒有覆蓋 `GEMINI.md` 的 Change Control、安全限制、Secrets 保護與檔案修改前確認流程。
* 是否沒有寫死不必要的 MCP Server alias。
* 是否仍以工具能力描述 MCP 使用方式，而不是綁定單一工具名稱。
* 是否已區分目標專案輸出與 `skills-hub` 自身輸出。
* 是否沒有要求 Gemini 在未經同意前建立、修改、刪除或搬移檔案。
* 是否沒有要求 Gemini 讀取與任務無關的敏感資訊。
* 是否沒有留下模板 placeholder 或未完成內容。
* 是否已在調整後執行 `/skills reload` 與 `/skills list` 確認載入結果。
